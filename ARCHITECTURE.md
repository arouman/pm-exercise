## Backend conventions

### Tech choices

- **Express 4** — battle-tested, near-zero learning curve, max AI training-data coverage.
- **Prisma** for ORM — the `schema.prisma` is your canonical data dictionary. Edit it, run `npx prisma migrate dev --name <change>`, restart. Generated client gives type-aware completions even in plain JS.
- **SQLite** for storage — file-based, no server, perfect for a take-home. SQLite **does not support Prisma enums**, so enum-like fields (`status`, `role`, `category`, `unit`, `type`) are declared `String` and validated in the route layer. Allowed values are documented in route Swagger annotations and listed in [AI-CONTEXT.md](./AI-CONTEXT.md).
- **Swagger** via `swagger-jsdoc` — annotations live next to the route they describe; the spec is built at startup and served at `/api/docs`.

### Route structure

Every resource file follows the same pattern:

```js
import { Router } from 'express';
import { prisma } from '../db.js';

const router = Router();

const VALID_X = [...];  // string-enum allowed values

/** @openapi ... */     // JSDoc annotation for Swagger
router.get('/', async (req, res, next) => { ... });
router.post('/', async (req, res, next) => { ... });
router.get('/:id', async (req, res, next) => { ... });
router.put('/:id', async (req, res, next) => { ... });

export default router;
```

Mount in [src/server.js](./src/server.js) with `app.use('/api/<resource>', router)`.

### Error handling

- Route handlers wrap their logic in `try/catch` and call `next(err)`. The global error handler at the bottom of `server.js` returns `{ error, details? }` as JSON with an appropriate status code.
- Known Prisma errors (`P2002` unique, `P2003` FK, `P2025` not found) are mapped to 404/409/400 explicitly in routes — the global handler is the catch-all.
- 400 for client mistakes (missing fields, invalid enum values), 404 for not-found, 409 for unique conflicts.

### Pagination & filtering

- List endpoints accept `?limit=` (default & max documented per route) and `?offset=` query params.
- Filter params are explicit: `?status=`, `?projectId=`, `?vendorId=`, etc. Look at the Swagger annotation on each route for the supported filters.
- Lists return a flat array, not a `{data, total}` envelope — kept simple on purpose.

### Naming

- Route paths: kebab-case (`/api/purchase-orders`).
- Prisma models: PascalCase singular (`PurchaseOrder`).
- Prisma fields: camelCase (`expectedDelivery`, `lineTotal`).
- JSON request/response keys: camelCase, mirror Prisma model names exactly.

## Frontend conventions

### Tech choices

- **React 18 + Vite** — fast cold start, no Webpack config, HMR by default.
- **JavaScript + JSX** (not TypeScript) — to match backend simplicity and keep the barrier low.
- **MUI v5** for components — extensive, well-documented, and AI tools know it cold.
- **React Router v6** for routing — URL-per-page.
- **TanStack Query** for data fetching — automatic loading/error states, request caching, refetch-on-mount. No global state management beyond that.
- **MUI X DataGrid Community** for entity list pages — sortable/filterable out of the box. For nested tables inside drawers, use plain `<Table>`.

### Folder layout

- `pages/` — top-level routes. One file per page.
- `components/` — anything reused across pages. Keep these dumb and styled via the theme.
- `hooks/` — react-query hooks. Use `useEntities(path)` and `useEntity(path)` as your defaults.
- `api/client.js` — single fetch wrapper. Don't import `fetch` directly in pages.

### When to add a page vs a drawer

- **Page** when the URL should be shareable, the content is the primary view, and there's substantial navigation/filtering needed (entity list, dashboard).
- **DetailDrawer** when the content is a sub-record or detail-of-a-row (PO line items, delivery items, project team list, inventory transactions). Drawers are easier on the user (they stay in context) and easier on you (no extra route, no separate fetch state for the parent list).

If you find yourself wanting to "drill down two levels," consider whether one of those levels should be inside the drawer.

### Theme & components

See [DESIGN-SYSTEM.md](./DESIGN-SYSTEM.md) for the full inventory. Quick reference:

- Always wrap pages with `<PageHeader title="..." subtitle="..." />`.
- Use `<StatusChip value="..." />` for any of: PO statuses, Delivery statuses, Project statuses, Transaction types.
- Use `<DetailDrawer>` for the row-click-to-see-more pattern.
- For dashboard metrics use `<KpiCard label="..." value={...} tone="success|warning|danger|neutral" />`.

## Dev experience

The root `package.json` orchestrates everything:

- `npm install` — runs root install, then via `postinstall`: client install → `prisma generate` → `prisma migrate deploy` → seed. One command.
- `npm start` — `concurrently` runs Express on `:3000` and Vite on `:5173`. Vite proxies `/api` to Express.
- `npm run db:reset` — wipes and reseeds. Idempotent.
- `npm run build && npm run start:prod` — production-style: Vite builds to `client/dist`, Express serves it on `:3000`.

The `dev.db` SQLite file is gitignored and regenerated on every fresh install.

## Out of scope (intentional non-goals)

- **No authentication / authorization.** Every request implicitly acts as anonymous. If your rental design depends on per-user permissions, mock them with a URL param or a select-user dropdown in the UI; don't try to bolt on real auth.
- **No real-time.** REST only. If you need live updates, poll with react-query (`refetchInterval`).
- **No file uploads.** If your rental design needs document attachments (rental agreements, damage photos), describe the integration in your write-up; don't build it.
- **No tests.** You're welcome to add them, but the starter ships without any.
- **No mobile-native code.** The web app is responsive; if your rental flows are field-first, design for mobile-web, not for a separate native app.
- **No external integrations.** No QuickBooks, no Procore, no ERP connectors. Reference them in your strategy doc if relevant; don't build them.

## Extending for rentals — high-level pointers

Rentals is **your design** — there's no canonical answer in this starter. But here are the integration seams already in the codebase:

- **`Project`** is the spatial anchor: equipment lives at a project (or is in transit between projects).
- **`User`** already has a `FLEET_MANAGER` role and seeded user (`Robin Castillo`) — use it.
- **`InventoryTransaction`** has a `TRANSFER` type already in the schema; this is a natural model for equipment moves between sites. Your rental code can layer on top.
- **`Vendor`** has a `TOOLS_RENTAL_EXTERNAL` category — these are the external rental suppliers in seed data, foreshadowing the procurement-from-external-rental-vendor flow.
- **API patterns** — copy any existing route file (e.g., `src/routes/purchase-orders.js`) when creating new ones. Don't reinvent the error/pagination conventions.
- **UI patterns** — copy any existing page (e.g., `client/src/pages/InventoryPage.jsx`) for a list-with-drawer-detail. Reuse `StatusChip`, `KpiCard`, `DetailDrawer`, `PageHeader`.

That's it. Build the rental product you'd want to use.
